# Provisioning Runbook — ручной онбординг тенанта

**Дата:** 2026-06-06
**Статус:** MVP (ручной процесс; автоматизация — отдельный под-проект)
**Основа:** `docs/architecture/adr-001-data-plane-isolation.md` (1 VPS : 1 клиент, bare Windows, бот как worker без изменений).

> Назначение: пошаговый чек-лист, чтобы развернуть нового платящего клиента на выделенном VPS вручную, пока спрос не доказан и не оправдана автоматизация. Provider-agnostic — конкретный провайдер подставляется на шаге 1.

---

## Предусловия (один раз)

- [ ] Готов установочный пакет бота: доступ к репо (приватный) ИЛИ собранный архив релиза.
- [ ] Зафиксирована версия Python 3.11 (зависимости пиннуты — `requirements.txt`).
- [ ] Подготовлен шаблон `account` и шаблоны `streams.json` по тарифам (см. Приложение A/B).
- [ ] Решён способ **безопасной** передачи MT5-кредов клиента (НЕ почта/мессенджер открытым текстом; напр. одноразовая ссылка/секрет-менеджер).

---

## Онбординг нового тенанта

### 1. VPS
- [ ] Поднять Windows-VPS: ≥2 GB RAM, ≥2 vCPU, SSD, низкая латентность к серверу брокера клиента (forex-оптимизированные VPS предпочтительны). Бюджет ~$15–22/мес (заложено в unit-эконе).
- [ ] Включить авто-обновления безопасности; настроить RDP-доступ только с твоего IP; задать сложный пароль администратора.
- [ ] Записать: tenant_id, провайдер, IP/host, дата создания.

### 2. MT5
- [ ] Установить терминал MetaTrader 5 нужного брокера клиента.
- [ ] Запустить терминал, войти кредами клиента, убедиться: котировки идут, символы тарифа доступны (напр. XAUUSD/EURUSD), торговля разрешена счётом.
- [ ] Запомнить полный путь к `terminal64.exe` (нужен для `MT5_PATH`).

### 3. Получить креды клиента
- [ ] Безопасно получить `login` / `password` / `server` MT5 клиента.
- [ ] Подтвердить у клиента: тип счёта (реальный/демо), брокер, плечо.

### 4. Деплой бота
- [ ] Скопировать код бота на VPS (clone/архив) в рабочую папку (напр. `C:\mk\`).
- [ ] Создать venv: `py -3.11 -m venv .venv` → активировать.
- [ ] `pip install -r requirements.txt` (MetaTrader5 встанет по win32-маркеру; версии пиннуты).

### 5. Конфигурация тенанта
- [ ] Создать `account.py` с кредами клиента (формат — Приложение A). Файл gitignored — НЕ коммитить.
- [ ] Создать `streams.json` по тарифу клиента (Приложение B: Entry = 5 стратегий / 2 потока; Pro = все / 10 потоков). **Здесь реализуется gating плана.**
- [ ] Задать env (через системные переменные или `.env`/обёртку запуска):
  - `MT5_PATH` = полный путь к `terminal64.exe` (служебная сессия — см. `authenticator.py`).
  - `ADMIN_PASSWORD` = стартовый пароль дашборда клиента (или передать клиенту сгенерированный).
  - `JWT_SECRET` = уникальный секрет на тенанта (иначе создастся `.jwt_secret`).
  - `HOST` = `127.0.0.1` (публикация через reverse-proxy) или `0.0.0.0` при прямом доступе; `PORT` = напр. `8080`.
  - `TRUSTED_HOSTS` / `CORS_ORIGINS` — под домен/IP клиента.

### 6. Авто-старт (базовая надёжность)
- [ ] Завернуть `python main.py` в Windows-сервис (NSSM или `sc.exe`) ИЛИ Task Scheduler «при загрузке», с рабочей папкой = папка бота и активированным venv.
- [ ] Проверить: сервис стартует при ребуте VPS, перезапускается при падении (базовый restart-on-failure).
- [ ] (Продвинутый хардеминг — алертинг/health-checks/crash-recovery — отдельный reliability-слайс, не здесь.)

### 7. Верификация
- [ ] Дашборд открывается (логин админ-кредами), MT5 показывает `connected`.
- [ ] Потоки тарифа `enabled`; на дашборде идут индикаторы/сигналы.
- [ ] Прошёл хотя бы один цикл; `GET /performance` отдаёт ответ (трек начал писаться в `performance.db`).
- [ ] Перезагрузить VPS → бот сам поднялся (шаг 6).

### 8. Учёт
- [ ] Записать маппинг **tenant_id ↔ VPS(IP/host) ↔ тариф ↔ dashboard URL ↔ дата старта** (таблица/файл — будущий control plane #2/#3 заберёт это в БД).
- [ ] Передать клиенту: URL дашборда, логин/пароль, краткую инструкцию.

---

## Offboarding (просрочка/отмена)
- [ ] По сигналу биллинга (просрочка > grace-период): остановить сервис бота (торговля прекращается).
- [ ] Сохранить `performance.db` клиента (трек/история) при необходимости.
- [ ] По окончательной отмене: депровижн VPS, удалить креды клиента (152-ФЗ — удаление ПДн).
- [ ] Обновить учётную таблицу (статус тенанта).

## Приложение A — формат `account.py`

```python
class Account:
    account = {
        "login":    12345678,            # int — номер счёта MT5 клиента
        "password": "<пароль клиента>",  # торговый или investor (см. модель доступа)
        "server":   "BrokerName-Server", # сервер брокера клиента
    }
```
⚠️ gitignored (паттерн `account*`). Хранить с шифрованием at-rest; не логировать.

## Приложение B — `streams.json` по тарифам

Структура потока (см. `streams.py` / `TradingStream`): `id, name, strategy, symbol, timeframe, volume, sl_atr, tp_atr, magic, enabled`. `magic` назначается реестром автоматически.

- **Entry ($59):** до **2 потоков**, выбор из **5 стратегий** (подмножество `STRATEGIES`).
- **Pro ($99):** до **10 потоков**, **все** стратегии.

Шаблоны держать рядом с runbook (`streams.entry.json` / `streams.pro.json`) и копировать как `streams.json` под клиента, проставляя его символы/объёмы. Лимиты тарифа на этом этапе соблюдаются **вручную при создании файла**; программный gate — отдельный под-проект (#5).

## Известные ограничения MVP-процесса (осознанные)
- Полностью ручной → не масштабируется за ~десятки клиентов (ок для фазы валидации).
- Единого портала нет: у каждого клиента свой dashboard на своём VPS.
- Лимиты тарифа enforced вручную (через `streams.json`), не кодом.
- Передача/хранение кредов — ручная дисциплина; программное шифрование/хранилище — под-проект #3.

Эти ограничения сознательны: автоматизацию строим только после доказанного спроса (см. ADR-001 «Последствия» и unit-economics «валидация прежде постройки»).
