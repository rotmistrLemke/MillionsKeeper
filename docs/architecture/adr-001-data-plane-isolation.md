# ADR-001 — Изоляция data plane для мультитенантного SaaS

**Дата:** 2026-06-06
**Статус:** принято
**Контекст-трек:** монетизация SaaS, под-проект #1 (load-bearing). См. `docs/business/2026-06-06-unit-economics-honest.md`.

## Контекст

MillionsKeeper движется к SaaS-модели: множество платящих клиентов, у каждого — своя торговля на собственном MT5-счёте. Нужно решить, **как изолировать торговлю N клиентов**.

Жёсткие факты:

- **MetaTrader5 Python-пакет (`mt5`) — процессный синглтон.** `mt5.initialize()` подключает ОДИН терминал/счёт на процесс; `mt5.shutdown()` рвёт IPC всему процессу. Это явно задокументировано в `main.py:39-42` (локальную переменную нельзя называть `auth`, иначе GC дёрнет `__del__`→`shutdown()` и все агенты получат `-10004 No IPC connection`).
- **Вся текущая архитектура — процесс-глобальный single-tenant:** `mt5`, `streams.registry`, `market_data_cache.cache`, `trading_status.status`, `account.Account.account` — module-level синглтоны. Один `main.py` = один счёт, один EventBus, 9 агентов, один dashboard.
- **Решение по хостингу уже принято** (unit-economics): оператор хостит, **1 VPS : 1 клиент** (изоляция кредов + надёжность).

Следствие: **код-уровневая мультитенантность в одном процессе невозможна** (два счёта в одном процессе нельзя). Изоляция обязана быть на уровне процесса/ОС.

## Решение

**Каждый тенант = один выделенный bare Windows-VPS, на котором работает существующий бот (`python main.py`) против MT5-счёта этого клиента с локальной конфигурацией.**

- Изоляция проходит по **границе VPS**, а не внутри кода.
- **Торговый бот не меняется** — он уже single-tenant; на каждом VPS крутится одна копия с локальными `account`/`streams.json`/`.jwt_secret`/`performance.db`/anomaly-db этого клиента.
- **Control plane** (отдельные под-проекты #2–#6) провижнит и управляет N таких VPS-воркеров; data plane сам по себе остаётся тем, что есть.

## Рассмотренные альтернативы (отвергнуты)

| Вариант | Почему отвергнут |
|---|---|
| Несколько счетов в одном процессе | Невозможно: `mt5` — процессный синглтон. |
| Процесс-на-тенанта на ОБЩЕМ VPS | Каждый MT5-терминал — тяжёлый Windows-GUI-процесс (RAM/CPU); конфликты IPC/сессий; противоречит выбранной 1:1-изоляции кредов. |
| Docker-контейнер на тенанта | MT5 — Windows-GUI-приложение; контейнеризация Windows-GUI болезненна и хрупка. |
| Код-мультитенантность (рефактор синглтонов в per-tenant контекст) | Не снимает MT5-ограничение (всё равно нужен процесс на счёт); огромный рефактор боевого пути ради нулевого выигрыша. |

## Последствия

**Положительные:**
- Минимальные изменения боевого кода (≈ноль) — риск для денежного пути отсутствует.
- Полная изоляция: сбой/компрометация одного тенанта не затрагивает других.
- Простая ментальная модель: «один клиент = один VPS = одна копия бота».

**Отрицательные / обязательства:**
- **Ops-heavy масштабирование:** N клиентов = N VPS; ручной провижнинг на старте (runbook), автоматизация — отдельный под-проект, и только после доказанного спроса.
- **Per-tenant стоимость = полный VPS** (~$15–22/мес) — учтено в unit-экономике (entry-тариф $59 right-sized под это).
- **Хранение чужих MT5-паролей** на VPS оператора — риск/обязательство (152-ФЗ, шифрование at-rest, безопасная передача). Прорабатывается в под-проекте аккаунтов (#3).
- **Dashboard per-VPS:** у каждого тенанта свой dashboard на своём VPS (его IP/домен). Единый портал — задача control plane (#6), не data plane.
- **Подписка ↔ воркер:** просрочка оплаты → control plane останавливает VPS-воркер (gating, #5).

## Граница этого решения

Этот ADR фиксирует ТОЛЬКО изоляцию data plane. Не входит и решается отдельными под-проектами: автоматизированный провижнинг, control-plane аккаунты/биллинг/gating, единый портал, reliability-хардеминг воркера. Ручной онбординг первых клиентов — в `docs/ops/provisioning-runbook.md`.
