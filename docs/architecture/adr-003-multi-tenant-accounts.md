# ADR-003 — Мультитенант-аккаунты: два слоя идентичности

**Дата:** 2026-06-06
**Статус:** принято (на бумаге; реализация — после валидации)
**Контекст-трек:** монетизация SaaS, под-проект #3. Зависит от ADR-001 (изоляция) и ADR-002 (control plane в Postgres).

## Контекст

Текущая аутентификация (`auth.py`) — это **дашборд одного деплоя**: роли `admin`/`user`, `users.json`, JWT (HS256), bootstrap-админ. Она защищает веб-панель ОДНОГО экземпляра бота, а не изолирует платящих клиентов.

SaaS нужен слой **per-customer аккаунтов** (тенантов): регистрация, email, тариф, подписка, привязка к своему VPS. Вопрос: как соотносятся новый клиентский аккаунт и существующий дашборд-auth, учитывая ADR-001 (каждый клиент — свой VPS с локальным дашбордом).

## Решение

**Две раздельные плоскости идентичности:**

1. **Control-plane аккаунт (tenant)** — платящий клиент.
   - Живёт в Postgres (`tenants`, ADR-002): email, password_hash, тариф, подписка, привязка к VPS.
   - Аутентификация — на **клиентском портале** (будущий #6): регистрация/логин, /account/billing, смена тарифа, ссылка на Customer Portal провайдера.
   - Это «кто платит и каким планом владеет».

2. **Per-VPS дашборд-auth** (`auth.py`, без изменений) — управление ботом на конкретном VPS клиента.
   - Остаётся локальным на каждом воркере: роли admin/user, JWT, `users.json`.
   - Креды генерируются control plane при провижнинге (provisioning-runbook шаг 5) и передаются клиенту.
   - Это «кто рулит этим конкретным экземпляром бота».

**Связь:** `tenant.id` → `vps_instances` (ADR-002) → `dashboard_url` + сгенерированные креды. Один тенант → один VPS (1:1, ADR-001).

## Почему не один слой

- **Слить в один (только control-plane auth, проксировать дашборд):** потребовало бы централизованного reverse-proxy/SSO к каждому VPS-дашборду и переписывания `auth.py` — большой объём, противоречит «воркер не меняется» (ADR-001). Отложено до единого портала (#6), и даже тогда дашборд-auth на воркере остаётся как fallback.
- **Только дашборд-auth (без control plane):** не даёт биллинга/подписок/мультиклиентности — не SaaS.

## Модель доступа MT5-кредов

- Клиент передаёт **investor (read-only)** или торговый MT5-пароль при онбординге (зависит от того, нужна ли реальная торговля или только мониторинг — для торговли нужен торговый).
- **Хранение:** торговый пароль шифруется at-rest. Варианты: (а) только на VPS-воркере (в `account`, доступ по диску VPS) — минимизирует поверхность; (б) в control plane зашифрованным (KMS/Fernet-ключ вне БД) для авто-провижнинга. **Решение MVP: на воркере** (control plane хранит лишь метаданные `mt5_login`, не пароль) — меньше риск/ответственность (152-ФЗ). Авто-провижнинг с шифрованным хранением в control plane — позже, при автоматизации.

## Переиспользование

- `auth.py` (JWT/bcrypt/роли) — **переиспользуется как есть** для дашборда воркера. Не переписываем.
- Control-plane auth — **новый** код (FastAPI control plane), может переиспользовать те же примитивы (passlib/jose), но отдельная БД-модель (`tenants`).

## Последствия

- **+** Чёткое разделение: «биллинговый аккаунт» ≠ «оператор дашборда»; каждый слой простой и независимый.
- **+** Нулевой риск для существующего `auth.py`/воркера.
- **−** Клиент имеет ДВЕ точки входа: клиентский портал (биллинг) и свой дашборд (управление ботом). Единый SSO — задача #6 (опционально).
- **−** Креды дашборда генерируются вручную при провижнинге (MVP) — автоматизация позже.
