# Тест-харнесс стратегий — дизайн

**Дата:** 2026-05-31
**Статус:** дизайн одобрен, готов к плану реализации
**Слайс:** A («надёжность ядра», шаг 1 из 3: A → рефактор settings.py → декомпозиция backtest.py)

---

## Цель и контекст

Торговые стратегии в [strategies/](../../../strategies/) напрямую генерируют сигналы, двигающие реальные деньги, но не покрыты тестами (в [tests/](../../../tests/) есть только legacy-тесты alligator/MA/MACD, и те **не собираются** — см. ниже). Прежде чем рефакторить `settings.py` (слайс B) и декомпозировать `backtest.py` (слайс C), нужна страховочная сетка, фиксирующая текущее поведение стратегий.

**Двойная цель тестов:**
1. **Сетка перед рефакторингом** — запереть текущее поведение, ловить регрессии при B/C.
2. **Валидация корректности** — доказать, что каждая стратегия даёт ожидаемый сигнал на намеренных сценариях.

**Среда выполнения:** тесты должны проходить на любой машине без живого MT5-терминала (CI-ready).

---

## Ключевые факты, на которых построен дизайн

- **Контракт стратегии** ([strategies/base.py](../../../strategies/base.py)): `compute_indicators(df) → df`, `get_entry_signal(row) → 'BUY'|'SELL'|None`, `get_exit_signal(row, position) → bool`, `get_sl_tp(row, signal, point) → (sl, tp)`. Плюс хуки: `wants_hedge`, `closes_on_weekend`, `uses_trailing_exit`, `get_hedge_exit_signal`, `on_trade_closed`, `indicator_columns`. Встроенный флэт-гард: `compute_flat_indicators` + `is_flat(row)` (2 из 3: ADX<20, BB-сжатие, ATR ниже среднего; NaN трактуется как флэт → не торгуем).
- **Реестр**: 20 стратегий в [strategies/__init__.py](../../../strategies/__init__.py) (`STRATEGIES`).
- **Импорты стратегий чистые**: модули стратегий зависят только от `talib` + `pandas`, **не импортируют** `settings`/`MetaTrader5`/`trading`. → тесты стратегий импортируются без стаба MT5.
- **Стратегии с состоянием**: `MacdHistStrategy._blocked_side`. [strategies/runtime.py](../../../strategies/runtime.py) кэширует singleton-экземпляры для live — тесты должны инстанцировать свежие объекты, минуя `runtime`.
- **Поломка legacy-тестов**: [settings.py:18](../../../settings.py#L18) читает `mt5.TIMEFRAME_M1` на уровне тела модуля; в окружении без полноценного MT5 это `AttributeError` при импорте → 3 теста (`test_alligatorbot`, `test_macd_atr`, `test_moving_average`) не собираются.
- **pytest**: `asyncio_mode = auto` ([pytest.ini](../../../pytest.ini)); [conftest.py](../../../conftest.py) добавляет корень проекта в `sys.path`.

## Архитектура и структура файлов

```
tests/
├── conftest.py                  # +MT5 sys.modules стаб (чинит сбор legacy-тестов)
├── strategies/
│   ├── __init__.py
│   ├── builders.py              # синтетические OHLC: trend_up/trend_down/flat/from_closes
│   ├── golden_utils.py          # прогон стратегии → сигнальная серия; сравнение/регенерация
│   ├── conftest.py              # фикстуры: real_ohlc (CSV), параметризация по STRATEGIES
│   ├── test_contract.py         # параметр. по всем 20 — контракт BaseStrategy
│   ├── test_golden.py           # параметр. по всем 20 — снимок сигналов на реальном CSV
│   └── test_behavioral.py       # по стратегии: сценарий → ожидаемый сигнал
├── golden/
│   └── <strategy>.json          # зафиксированные сигнальные серии
└── fixtures/
    └── xauusd_h1.csv            # ~500 реальных баров XAUUSD H1 (разовый забор)

tools/
└── dump_ohlc.py                 # разовый скрипт забора баров из MT5 → CSV (локально)
```

Каждый юнит изолирован: `builders.py` строит данные, `golden_utils.py` прогоняет и сравнивает, тестовые файлы только описывают сценарии и ассерты.

## Источники данных (гибрид)

### Синтетика — `builders.py`
Детерминированные (фикс. seed) `pandas.DataFrame` с колонками `time, open, high, low, close, tick_volume`.
- Длина по умолчанию **≥250 баров** (достаточно для EMA200 и flat-avg периода 50).
- Паттерны: `trend_up(n)`, `trend_down(n)`, `flat(n)` (узкий диапазон → флэт-гард активен), `from_closes(closes)` (кастомная серия close, OHLC достраивается консистентно: `high≥max(open,close)`, `low≤min(open,close)`).
- Семантика colon: `time` — равномерная сетка H1 (`pandas.date_range`).

### Реальные данные — `fixtures/xauusd_h1.csv`
- ~500 реальных баров XAUUSD H1, снятых **разово** скриптом `tools/dump_ohlc.py` (требует MT5, запускается локально).
- Коммитится в репо. Тесты только читают → CI не нужен MT5.

## Категории тестов

### A1. Контракт (параметризовано ×20)
Для каждой стратегии из `STRATEGIES`:
- `get_entry_signal(row)` возвращает строго `'BUY'`, `'SELL'` или `None`.
- `get_sl_tp(row, 'BUY', point)`: при включённых SL/TP — `sl < entry_close < tp`; для `'SELL'` — `tp < entry_close < sl`. Если SL или TP отключён (значение 0/None по логике стратегии) — соответствующая сторона не проверяется на неравенство, но тип результата валиден (кортеж из 2 элементов).
- `compute_indicators(df)` не кидает исключение и не уменьшает число строк df.
- `is_flat(row)` возвращает `bool`.
- Хуки `wants_hedge()`, `closes_on_weekend()`, `uses_trailing_exit()`, `get_hedge_exit_signal(...)` возвращают `bool`.
- Робастность: на коротком df (< периода индикатора) и на df с NaN — ни `compute_indicators`, ни `get_entry_signal` не кидают исключение (NaN-строки дают `None`/флэт).

### A2. Golden-снимки (параметризовано ×20)
- Прогон стратегии по `xauusd_h1.csv`: `compute_indicators` + `compute_flat_indicators`, затем построчно собрать серию входных сигналов `[(idx, 'BUY'|'SELL'), ...]` (только не-`None`, с учётом флэт-гарда) и серию exit-флагов.
- Сериализация в `tests/golden/<strategy>.json`; строгое сравнение с зафиксированным файлом.
- Регенерация — кастомная pytest-опция `--update-golden` (при ней тест перезаписывает golden-файл и помечается xfail/skip с пометкой «updated», а не сравнивает).
- Первый запуск: golden-файлы генерируются и коммитятся как baseline текущего поведения.

### A3. Поведенческие (по стратегии, отдельный под-этап)
Для каждой из 20 стратегий: ≥1 сценарий «должен войти» (синтетика, построенная под её логику входа, выведенную чтением `get_entry_signal`) + сценарий «флэт → `None`».
- Реализуется через TDD: вчитываемся в логику входа конкретной стратегии, строим минимальный сценарий, утверждаем ожидаемый сигнал.
- Выделено в отдельный под-этап **после A1+A2**, чтобы сетка (контракт + golden) заработала максимально быстро; поведенческие добиваются инкрементально.

## MT5-стаб (бонус, в корневом conftest)

В корневом [tests/conftest.py](../../../tests/conftest.py) (или дополнении к корневому [conftest.py](../../../conftest.py)) — вставка фейкового модуля `MetaTrader5` в `sys.modules` **только если** реальный модуль отсутствует/неполноценен (нет атрибута `TIMEFRAME_M1`). Стаб предоставляет:
- константы `TIMEFRAME_M1/M5/M15/M30/H1/H4/D1` (целые-заглушки),
- no-op/`None`-возвращающие заглушки функций, к которым обращаются `settings.py`/`indicators.py` на уровне импорта.

Назначение: чинит сбор 3 legacy-тестов и страхует golden-путь, если он когда-нибудь подтянет `settings`. Не трогает продакшен-код (рефактор `settings.py` — слайс B).

## Краевые случаи и обработка ошибок

- **Состояние стратегий**: каждый тест инстанцирует свежий объект `StrategyClass()`, не через `runtime.get_runtime_strategy` (иначе `_blocked_side` протекает между тестами).
- **NaN от talib на коротких сериях**: контракт-тест явно проверяет отсутствие исключений; флэт-гард уже трактует NaN как «не торгуем».
- **Отключённые SL/TP** (значение 0): контракт-тест не требует неравенства для отключённой стороны.
- **Детерминизм**: синтетические билдеры используют фиксированный seed; golden — фиксированный CSV. Тесты не зависят от текущего времени/рынка.

## Объём и границы

**Входит:** инфраструктура (builders, golden_utils, фикстуры, CSV, MT5-стаб), контракт-тесты (A1), golden-тесты (A2), поведенческие тесты (A3).
